# Requirements Document

## Introduction

This document defines the requirements for migrating the COUNCIL multi-agent AI editorial boardroom from static HTML prototypes to a production-ready Next.js application with fully serverless AWS backend infrastructure. COUNCIL orchestrates structured debates between AI personas (Hype-Man, Professor, Skeptic) before generating content, with users acting as "Directors" who guide the debate and select strategic paths.

## Glossary

- **COUNCIL_System**: The complete multi-agent AI editorial boardroom platform including frontend and backend components
- **Landing_Page**: The marketing and entry page for the COUNCIL platform
- **Session_Workspace**: The interactive debate interface where users orchestrate AI agent discussions
- **AI_Agent**: An individual AI persona (Hype-Man, Professor, Skeptic, or Mediator) with specialized reasoning patterns
- **Debate_Engine**: The backend orchestration system that manages parallel AI agent interactions via Amazon Bedrock
- **Strategic_Path**: One of three synthesized directions (Aggressive, Balanced, Technical) generated by the Mediator
- **Annotated_Content**: Final generated content with reasoning tooltips explaining decision rationale
- **Session_Manager**: The system component responsible for tracking debate state and user interactions
- **Storage_Layer**: The combined S3 and DynamoDB infrastructure for persisting debate data
- **API_Gateway**: AWS API Gateway providing REST and WebSocket endpoints for client-server communication
- **Lambda_Orchestrator**: AWS Lambda functions coordinating the multi-agent workflow
- **Bedrock_Client**: The interface layer for invoking Amazon Bedrock AI models
- **Real_Time_Stream**: WebSocket or Server-Sent Events connection delivering live debate updates to clients

## Requirements

### Requirement 1: Frontend Migration to Next.js

**User Story:** As a developer, I want to migrate the existing HTML pages to Next.js, so that the application has a modern React-based architecture with server-side rendering capabilities.

#### Acceptance Criteria

1. THE COUNCIL_System SHALL convert landing_page.html into a Next.js page component at /app/page.tsx
2. THE COUNCIL_System SHALL convert session_workspace.html into a Next.js page component at /app/session/page.tsx
3. THE COUNCIL_System SHALL preserve all existing Tailwind CSS styling and dark mode functionality
4. THE COUNCIL_System SHALL maintain the Inter font family and Material Symbols icon integration
5. THE COUNCIL_System SHALL extract reusable UI elements into React components (Header, AgentCard, DebateMessage, PathSelector)
6. THE COUNCIL_System SHALL implement the canvas animation as a client-side React component
7. THE COUNCIL_System SHALL configure Next.js with Tailwind CSS and the existing color palette (primary: #E8734A, background-dark: #0A0A0A)
8. THE COUNCIL_System SHALL implement page transitions using Next.js navigation instead of window.location redirects

### Requirement 2: UI Component Library Integration

**User Story:** As a developer, I want to integrate ShadCN UI and Framer Motion, so that the application has accessible, animated components following modern design patterns.

#### Acceptance Criteria

1. THE COUNCIL_System SHALL install and configure ShadCN UI component library
2. THE COUNCIL_System SHALL replace custom button implementations with ShadCN Button components where appropriate
3. THE COUNCIL_System SHALL integrate Framer Motion for page transitions and debate message animations
4. THE COUNCIL_System SHALL implement smooth entrance animations for debate messages using Framer Motion
5. THE COUNCIL_System SHALL maintain WCAG 2.1 AA accessibility standards for all interactive components
6. WHERE custom styling conflicts with ShadCN defaults, THE COUNCIL_System SHALL override with the existing COUNCIL design tokens

### Requirement 3: AWS Serverless Backend Architecture

**User Story:** As a system architect, I want to establish a fully serverless AWS infrastructure, so that the application scales automatically without managing servers.

#### Acceptance Criteria

1. THE COUNCIL_System SHALL deploy all backend logic using AWS Lambda functions (no EC2 or container services)
2. THE COUNCIL_System SHALL configure Amazon API Gateway with REST endpoints for session management
3. THE COUNCIL_System SHALL configure Amazon API Gateway with WebSocket endpoints for real-time debate streaming
4. THE COUNCIL_System SHALL use Amazon S3 for storing debate transcripts and session logs
5. THE COUNCIL_System SHALL use Amazon DynamoDB for storing session metadata and user selections
6. THE COUNCIL_System SHALL configure Amazon CloudWatch for logging all Lambda invocations and errors
7. THE COUNCIL_System SHALL define IAM roles with least-privilege access for Lambda execution
8. THE COUNCIL_System SHALL enable CORS on API Gateway endpoints to allow Next.js frontend requests

### Requirement 4: Amazon Bedrock Integration

**User Story:** As a developer, I want to integrate Amazon Bedrock AI models, so that the system can invoke Claude and Llama models for agent debates.

#### Acceptance Criteria

1. THE Bedrock_Client SHALL invoke Claude 3.5 Sonnet for the Professor agent
2. THE Bedrock_Client SHALL invoke Claude 3.5 Sonnet for the Skeptic agent
3. THE Bedrock_Client SHALL invoke Claude 3.5 Sonnet for the Mediator agent
4. THE Bedrock_Client SHALL invoke Meta Llama 3 70B Instruct for the Hype-Man agent
5. THE Bedrock_Client SHALL configure appropriate temperature and max_tokens parameters for each agent persona
6. THE Bedrock_Client SHALL handle Bedrock API throttling errors with exponential backoff retry logic
7. WHEN a Bedrock invocation fails, THE Bedrock_Client SHALL log the error to CloudWatch and return a descriptive error message
8. THE Bedrock_Client SHALL support streaming responses from Bedrock models for real-time debate display

### Requirement 5: Multi-Agent Debate Orchestration

**User Story:** As a user, I want to submit a prompt and watch three AI agents debate in real-time, so that I can see multiple perspectives before making a decision.

#### Acceptance Criteria

1. WHEN a user submits a prompt, THE Debate_Engine SHALL invoke three AI agents (Hype-Man, Professor, Skeptic) in parallel
2. THE Debate_Engine SHALL configure each agent with persona-specific system prompts defining their reasoning bias
3. THE Debate_Engine SHALL stream each agent's response to the frontend via Real_Time_Stream as tokens are generated
4. THE Debate_Engine SHALL enforce a maximum debate duration of 180 seconds per agent
5. THE Debate_Engine SHALL store the complete debate transcript in S3 with the session ID as the key prefix
6. WHEN all three agents complete their responses, THE Debate_Engine SHALL trigger the Mediator agent
7. IF any agent invocation fails, THE Debate_Engine SHALL continue with the remaining agents and log the failure

### Requirement 6: Mediator Synthesis and Strategic Path Generation

**User Story:** As a user, I want the Mediator to synthesize the debate into three strategic paths, so that I can choose the direction that best fits my goals.

#### Acceptance Criteria

1. WHEN the debate phase completes, THE Debate_Engine SHALL invoke the Mediator agent with the full debate transcript
2. THE Mediator SHALL generate exactly three Strategic_Paths labeled as Path A, Path B, and Path C
3. THE Mediator SHALL ensure each Strategic_Path includes a title, description, and primary agent influence (Hype-Man, Professor, or Skeptic)
4. THE Mediator SHALL format the Strategic_Paths as structured JSON for frontend consumption
5. THE COUNCIL_System SHALL display the three Strategic_Paths in the Session_Workspace path selector panel
6. THE COUNCIL_System SHALL store the generated Strategic_Paths in DynamoDB linked to the session ID
7. THE Mediator SHALL complete synthesis within 60 seconds of receiving the debate transcript

### Requirement 7: User Path Selection and Final Content Generation

**User Story:** As a user, I want to select one of the three strategic paths and receive annotated final content, so that I understand the reasoning behind the generated output.

#### Acceptance Criteria

1. WHEN a user selects a Strategic_Path, THE Session_Manager SHALL record the selection in DynamoDB with a timestamp
2. THE Debate_Engine SHALL invoke the Mediator agent with the selected path and debate context
3. THE Mediator SHALL generate Annotated_Content with inline reasoning tooltips explaining key decisions
4. THE Annotated_Content SHALL include at least 3 reasoning annotations per 500 words of generated text
5. THE COUNCIL_System SHALL render reasoning annotations as hoverable tooltips in the Final Draft panel
6. THE COUNCIL_System SHALL format reasoning tooltips with the agent name, timestamp, and explanation text
7. THE COUNCIL_System SHALL store the final Annotated_Content in S3 with the session ID and selected path as metadata

### Requirement 8: Real-Time Debate Streaming

**User Story:** As a user, I want to see debate messages appear in real-time as agents generate them, so that I experience the live deliberation process.

#### Acceptance Criteria

1. THE COUNCIL_System SHALL establish a WebSocket connection between the frontend and API Gateway when a session starts
2. WHEN an AI_Agent generates tokens, THE Lambda_Orchestrator SHALL push message chunks through the WebSocket connection
3. THE Session_Workspace SHALL append incoming debate messages to the Live Debate panel with smooth animations
4. THE Session_Workspace SHALL display agent avatar, name, timestamp, and message content for each debate entry
5. THE Session_Workspace SHALL auto-scroll the Live Debate panel to show the most recent message
6. IF the WebSocket connection drops, THE COUNCIL_System SHALL attempt reconnection with exponential backoff up to 5 retries
7. THE COUNCIL_System SHALL display a connection status indicator (green pulse for active, red for disconnected)

### Requirement 9: Session State Management

**User Story:** As a user, I want my debate session to persist across page refreshes, so that I don't lose progress if I accidentally close the browser.

#### Acceptance Criteria

1. WHEN a user starts a session, THE Session_Manager SHALL generate a unique session ID (UUID v4 format)
2. THE Session_Manager SHALL store the session ID in the browser's localStorage
3. THE Session_Manager SHALL persist session state in DynamoDB including: prompt, debate status, selected path, and timestamps
4. WHEN a user returns to the Session_Workspace, THE COUNCIL_System SHALL retrieve the session state from DynamoDB using the stored session ID
5. THE COUNCIL_System SHALL restore the Live Debate panel with all previous messages from S3
6. THE COUNCIL_System SHALL restore the selected Strategic_Path and Final Draft content if previously generated
7. THE Session_Manager SHALL mark sessions as expired after 24 hours of inactivity

### Requirement 10: Debate Transcript Storage and Retrieval

**User Story:** As a system administrator, I want all debate transcripts stored in S3, so that we can analyze agent performance and user interactions over time.

#### Acceptance Criteria

1. THE Storage_Layer SHALL store each debate transcript as a JSON file in S3 with the key pattern: `debates/{session_id}/transcript.json`
2. THE Storage_Layer SHALL include metadata in the S3 object: session_id, user_id (if authenticated), timestamp, and agent model versions
3. THE Storage_Layer SHALL store the final Annotated_Content in S3 with the key pattern: `debates/{session_id}/final_draft.json`
4. THE Storage_Layer SHALL configure S3 lifecycle policies to transition objects to Glacier after 90 days
5. THE Storage_Layer SHALL enable S3 server-side encryption (SSE-S3) for all stored objects
6. THE Lambda_Orchestrator SHALL write to S3 asynchronously to avoid blocking the debate workflow
7. WHEN a transcript write fails, THE Lambda_Orchestrator SHALL log the error to CloudWatch and retry up to 3 times

### Requirement 11: Session Metadata Persistence

**User Story:** As a developer, I want session metadata stored in DynamoDB, so that we can quickly query session status without reading large S3 objects.

#### Acceptance Criteria

1. THE Session_Manager SHALL create a DynamoDB table named `council_sessions` with `session_id` as the partition key
2. THE Session_Manager SHALL store the following attributes: session_id, user_prompt, debate_status, selected_path, created_at, updated_at
3. THE Session_Manager SHALL update the `debate_status` field as the session progresses through states: INITIATED, DEBATING, MEDIATING, PATH_SELECTED, COMPLETED
4. THE Session_Manager SHALL index the `created_at` field to enable time-based queries
5. THE Session_Manager SHALL configure DynamoDB with on-demand billing mode for automatic scaling
6. THE Session_Manager SHALL implement optimistic locking using a `version` attribute to prevent concurrent update conflicts
7. WHEN a DynamoDB write fails due to throttling, THE Session_Manager SHALL retry with exponential backoff up to 3 attempts

### Requirement 12: Error Handling and Monitoring

**User Story:** As a system administrator, I want comprehensive error logging and monitoring, so that I can diagnose issues and maintain system reliability.

#### Acceptance Criteria

1. THE COUNCIL_System SHALL log all Lambda function invocations to CloudWatch with request ID, duration, and memory usage
2. WHEN a Bedrock API call fails, THE Lambda_Orchestrator SHALL log the error with model name, request parameters, and error message
3. WHEN a DynamoDB operation fails, THE Session_Manager SHALL log the error with table name, operation type, and item key
4. WHEN an S3 operation fails, THE Storage_Layer SHALL log the error with bucket name, key, and operation type
5. THE COUNCIL_System SHALL configure CloudWatch alarms for Lambda error rates exceeding 5% over a 5-minute period
6. THE COUNCIL_System SHALL configure CloudWatch alarms for API Gateway 5xx errors exceeding 10 requests per minute
7. THE COUNCIL_System SHALL implement structured logging with JSON format including: timestamp, log_level, component, message, and context

### Requirement 13: Authentication and Authorization (Optional for MVP)

**User Story:** As a user, I want to authenticate with my account, so that my debate sessions are private and associated with my profile.

#### Acceptance Criteria

1. WHERE authentication is enabled, THE COUNCIL_System SHALL integrate AWS Cognito for user management
2. WHERE authentication is enabled, THE COUNCIL_System SHALL require JWT tokens for all API Gateway requests
3. WHERE authentication is enabled, THE Session_Manager SHALL associate sessions with the authenticated user_id
4. WHERE authentication is enabled, THE COUNCIL_System SHALL restrict session access to the owning user
5. WHERE authentication is enabled, THE COUNCIL_System SHALL implement a login page using Next.js and Cognito Hosted UI

### Requirement 14: Frontend API Client

**User Story:** As a developer, I want a typed API client in the Next.js frontend, so that backend communication is type-safe and maintainable.

#### Acceptance Criteria

1. THE COUNCIL_System SHALL implement a TypeScript API client module for all backend endpoints
2. THE API client SHALL define TypeScript interfaces for all request and response payloads
3. THE API client SHALL handle WebSocket connection lifecycle (connect, disconnect, reconnect)
4. THE API client SHALL implement request retry logic for transient network failures
5. THE API client SHALL expose methods: `startSession()`, `submitPrompt()`, `selectPath()`, `getSessionState()`
6. THE API client SHALL validate response schemas using Zod or similar runtime validation library
7. WHEN an API request fails, THE API client SHALL throw typed errors with status codes and error messages

### Requirement 15: Deployment and Infrastructure as Code

**User Story:** As a DevOps engineer, I want infrastructure defined as code, so that the AWS resources can be deployed consistently across environments.

#### Acceptance Criteria

1. THE COUNCIL_System SHALL define all AWS resources using AWS CDK or Terraform
2. THE infrastructure code SHALL parameterize environment-specific values (stage, region, resource names)
3. THE infrastructure code SHALL create separate stacks for development, staging, and production environments
4. THE infrastructure code SHALL output API Gateway endpoint URLs and WebSocket URLs after deployment
5. THE COUNCIL_System SHALL include a deployment script that builds the Next.js app and deploys Lambda functions
6. THE infrastructure code SHALL configure Lambda environment variables for S3 bucket names and DynamoDB table names
7. THE infrastructure code SHALL enable AWS X-Ray tracing for Lambda functions and API Gateway

### Requirement 16: Performance and Scalability

**User Story:** As a system architect, I want the system to handle concurrent users efficiently, so that the platform scales with user growth.

#### Acceptance Criteria

1. THE Lambda_Orchestrator SHALL complete debate orchestration within 180 seconds for 95% of requests
2. THE COUNCIL_System SHALL support at least 100 concurrent WebSocket connections per API Gateway stage
3. THE Bedrock_Client SHALL implement connection pooling to reuse HTTP connections across invocations
4. THE COUNCIL_System SHALL configure Lambda functions with appropriate memory allocation (minimum 512MB for Bedrock calls)
5. THE COUNCIL_System SHALL implement Lambda reserved concurrency limits to prevent runaway costs during traffic spikes
6. THE Session_Workspace SHALL render debate messages with less than 100ms latency from WebSocket receipt
7. THE COUNCIL_System SHALL lazy-load the canvas animation component to reduce initial page load time

### Requirement 17: Cost Optimization

**User Story:** As a product owner, I want to minimize AWS costs, so that the platform remains economically viable during early growth.

#### Acceptance Criteria

1. THE COUNCIL_System SHALL configure Lambda functions with ARM64 architecture (Graviton2) for 20% cost savings
2. THE Storage_Layer SHALL use S3 Intelligent-Tiering for automatic cost optimization of debate transcripts
3. THE Session_Manager SHALL configure DynamoDB with on-demand billing to avoid paying for unused capacity
4. THE COUNCIL_System SHALL implement Lambda function timeout limits (maximum 180 seconds) to prevent runaway executions
5. THE COUNCIL_System SHALL cache Bedrock model responses in DynamoDB for identical prompts within a 1-hour window
6. THE COUNCIL_System SHALL implement API Gateway request throttling (100 requests per second per user) to prevent abuse
7. THE infrastructure code SHALL tag all AWS resources with cost allocation tags (project: council, environment: {stage})

### Requirement 18: Testing and Quality Assurance

**User Story:** As a developer, I want automated tests for critical workflows, so that regressions are caught before production deployment.

#### Acceptance Criteria

1. THE COUNCIL_System SHALL include unit tests for all Lambda function handlers with minimum 80% code coverage
2. THE COUNCIL_System SHALL include integration tests for the complete debate workflow (prompt → debate → mediation → path selection)
3. THE COUNCIL_System SHALL mock Bedrock API calls in tests using fixtures with realistic agent responses
4. THE COUNCIL_System SHALL include end-to-end tests for the Next.js frontend using Playwright or Cypress
5. THE COUNCIL_System SHALL validate DynamoDB schema constraints in tests (required fields, data types)
6. THE COUNCIL_System SHALL test WebSocket connection handling including reconnection scenarios
7. THE COUNCIL_System SHALL run tests in CI/CD pipeline before allowing deployment to staging or production
